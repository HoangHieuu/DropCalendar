# SnapCal Documentation Map

## Product And Delivery

- `product/` — living SnapCal behavior derived from the source `SPEC.md`.
- `ARCHITECTURE.md` — macOS-first target architecture, provider boundaries,
  persistence, observability, and proof ladder.
- `stories/` — active work packets plus candidate phase/epic backlog.
- `TEST_MATRIX.md` — critical safety gates and proof vocabulary.
- `decisions/` — durable product, architecture, and Harness decisions.
- `DEVELOPMENT_CAPABILITIES.md` — available plugins, skills, tools, gaps, and
  phase-specific usage policy.
- `GLOSSARY.md` — shared product and Harness language.

## Harness Operations

- `HARNESS.md` — human-agent collaboration model.
- `FEATURE_INTAKE.md` — request type and risk classification.
- `CONTEXT_RULES.md` — bounded retrieval by lane and phase.
- `TOOL_REGISTRY.md` — capability registry protocol.
- `TRACE_SPEC.md` — trace and evidence requirements.
- `HARNESS_BACKLOG.md`, `HARNESS_AUDIT.md`, `HARNESS_MATURITY.md`,
  `HARNESS_COMPONENTS.md`, and `IMPROVEMENT_PROTOCOL.md` — maintenance and
  evolution of the Harness itself.
- `contracts/` — versioned orchestration contracts.
- `templates/` — story, decision, plan, and validation templates.

## Current State

The SnapCal contract is wired; application code is absent. `SPEC.md` is the
input snapshot, while product docs, selected stories, executable proof, and
accepted decisions are the living operating surface.
